# Matrix-run/v1 evidence contract

`matrix-run/v1` is a backend-only standalone artifact manifest for VM harness matrix evidence. It is not a daemon-event and not a `daemon-event/v1` extension, not an application client feature contract, not a simulator fixture, and not production or release approval.

The artifact schema string and JSON Schema `$id` are both `zig-scheduler/matrix-run/v1`.

## Required safety invariants

- `host_mutation` is always `false`; host rows may only prove refusal.
- `release_eligible` is always `false`; a matrix artifact is lab evidence only.
- Artifact paths are repository-relative data paths. Absolute paths and `..` traversal are invalid both in the standalone schema and in the checker. This applies to top-level artifact paths plus `policy.object_path`, `policy.source_path`, `workload.spec_path`, and `privacy_scan.report_path`.
- Manifest runs are rooted at `evidence/lab/matrix/<run-id>/manifest.json`; `<run-id>` and manifest `matrix_run_id` must be the same directory basename, 1-64 characters, using only `A-Z`, `a-z`, `0-9`, `_`, `.`, and `-`.
- `rollback_proof_path`, `cleanup_proof_path`, and `host_refusal_proof_path` are required for every outcome, including `SKIP` and `REFUSE`.
- `live-backend` is a named live scenario. A PASS live-backend manifest row must use `evidence_mode=vm-live`, include `daemon_event_path`, require a real VM marker fact with `required=true`, `present=true`, path `/run/zig-scheduler-vm-lab.marker`, and reference a row-local marker proof under the same matrix run root. The backend summary for a PASS live-backend row must point its `live_summary` field at the canonical `backend/live/summary.json` bundle, and that nested bundle must pass `qa/live_behavior_check.py` before the row can remain PASS. A noncanonical nested summary such as `observe-partial/summary.json` must never count as live PASS. The committed `fixtures/matrix-run/live-backend.json` is an inert fixture-only consumer contract sample and must not claim VM-live.
- Host-safe fixture PASS rows use `evidence_mode=fixture`, keep `vm_marker.required=false` and `vm_marker.present=false`, and remain `release_eligible=false`.
- Host-refusal-only rows use `evidence_mode=host-refusal-only`, keep `vm_marker.present=false`, and must not report `PASS`.
- Stale or dirty git state is rejected by the contract (`git.status` must be `current`, `expected_sha` must match `actual_sha`, and `dirty=false`).
- Privacy scans must pass with `private_fields_found=false`; docs and fixtures are inert data and must not execute external text.
- Workload rows may reference `benchmark-output/v1` records through row-local workload specs with `threshold_source=record-only` and `calibration_status=uncalibrated` until separately proven. These references are record-only provenance: the matrix checker verifies the referenced record hash and reuses `qa/benchmark_output_check.py` validation, but no benchmark record can create PASS/FAIL thresholds, release eligibility, production-capacity claims, or scheduler-performance claims.
- Todo 7 cgroup/hotplug workload specs carry VM-only semantic labels without expanding the BPF ABI: `workload-cgroup-weight-quota` requires `cpu.weight=callback-observed`, `cpu.max=deferred`, `cpu.max.burst=deferred`, `cpuset.cpus`/`cpuset.cpus.effective=observed-constraints`, uclamp and pressure `observed-or-deferred`, threaded/domain cgroups `observed`, and allowed-mask `rejected`; `workload-cpu-hotplug` requires offline/online CPU fallback `fallback-observed`, cpuset constraints observed, and allowed-mask rejection. These labels are contract evidence only and do not permit host cgroup, cpuset, affinity, priority, `/sys`, or `/proc` mutation.
- Matrix rows are pinned to `bpf_abi_version=zigsched-bpf-abi-v1`; ABI-v3 runtime sample metadata remains a separate evidence surface and must not be inferred from matrix fixtures.
- Protected-core suite validation is a companion gate (`qa/protected_core_suite_check.py`) layered after matrix validation: a protected PASS suite must contain `live-backend`, `workload-cpu-saturation`, `workload-cgroup-weight-quota`, and exactly one PASS row from `workload-interactive-latency` or `workload-scheduler-affinity-churn`; row-local artifacts must not be shared across rows; the cgroup row must link ABI-v3 runtime evidence proving `cpu.weight=callback-observed` and `cpu.max=deferred`. Protected-core runtime telemetry normalization is enforced by `qa/protected_core_telemetry_check.py`: every VM-live PASS row must expose sched_ext phase/state/root ops/enable_seq/task `ext.enabled`, events/`nr_rejected` coherence, policy and scheduler counters, sample-loss/backpressure, DSQ depth, queue latency, fairness/starvation, cgroup membership status/digest, redacted sched_ext observations, and record-only benchmark histogram references when present. Event counters may be numeric only when the source events are present; unavailable events require explicit unavailable counter facts rather than fake zeroes.
- VM-only sched_ext state rows cover before/during/after attach, normal unregister or revert, scheduler exit, watchdog/stall/error disable, and SysRq-S forced-disable observations when supported by the disposable VM. They must keep `host_mutation=false`, include host refusal plus rollback/cleanup paths, and encode scheduler state with `sched_ext`, `ops`, optional nondecreasing `enable_seq`, root ops, per-task `ext.enabled` availability, and disable reason labels. Raw `/sys`, `/proc`, command-line, environment, token, private debug dumps, stale `enable_seq`, negative counters, production/release/performance claims, or host-mutation claims are invalid.
- The manifest and nested contract objects are closed shapes (`additionalProperties=false`); extra unexpected properties are invalid even if all required fields are present.

## Outcomes

| Outcome | Meaning |
| --- | --- |
| `PASS` | VM-live lab row has complete marker, runtime, rollback, cleanup, host-refusal, privacy, and non-release evidence. |
| `SKIP` | Prerequisite was unavailable or unsupported; rollback, cleanup, and host-refusal proof still exist. |
| `REFUSE` | Fail-closed refusal evidence, usually host-refusal-only, with no host mutation. |
| `INCIDENT` | Unsafe or incomplete lab state captured as incident evidence. |
| `FAIL` | Matrix execution failed, including rollback failure or cleanup residue evidence. |

## Required fields

The v1 manifest requires matrix run ID, scenario ID, outcome, kernel tuple, supported tuple status, VM marker facts, BPF ABI version, policy object/hash metadata, workload spec/hash, action ID, audit ID, rollback ID, pre/post scheduler state (including sched_ext state, root ops, enable_seq, rollback/teardown labels where applicable), pre/post cgroup state, runtime sample path, daemon_event_path, incident path, rollback proof path, cleanup proof path, host refusal proof path, privacy scan, git state, `release_eligible=false`, and `host_mutation=false`. Workload specs carry canonical required tools, `threshold_source=record-only` for the committed workload rows, and may include `benchmark_provenance` entries with `record_path`, `record_sha256`, and `record_only=true`; in manifest mode each path must stay under the matrix run directory and must validate as `benchmark-output/v1`.

## Validation

Run:

```sh
python3 qa/matrix_run_contract_check.py --fixtures fixtures/matrix-run --schemas schemas/control --docs docs/control
```

The checker validates committed valid fixtures and confirms fixtures under `fixtures/matrix-run/invalid/` are rejected. Manifest mode additionally dereferences daemon events, runtime samples, incident evidence, rollback proof, cleanup proof, host refusal proof, workload capability/spec files, benchmark provenance records, and privacy reports instead of trusting path strings. The invalid fixtures cover malformed JSON, invalid outcome, stale and dirty git, missing VM marker on a VM-live row, forged live-backend PASS rows without row-root marker proof, absolute and traversing artifact paths, out-of-root daemon_event_path, missing rollback proof, missing cleanup proof, missing cleanup proof on both `SKIP` and `REFUSE`, missing host refusal proof, privacy failure, uncataloged workloads, required-tools/threshold-source/capability mismatches, wrong cgroup or CPU-hotplug semantic labels, unsupported matrix BPF ABI drift, private token or debug-dump leakage, stale sched_ext enable_seq, missing sched_ext state, release/production/performance claim leakage, spec-hash mismatches, missing/malformed/claim-bearing benchmark provenance in manifest artifacts, extra unexpected properties, `host_mutation=true`, and `release_eligible=true`.


Manual protected VM proof bundles wrap the matrix manifest in `zig-scheduler/evidence-manifest/v1` (`evidence-manifest.json`). That bundle-level evidence manifest is validated by `qa/evidence_manifest_check.py` with `schemas/control/evidence-manifest.v1.schema.json` and records SHA-256 hashes plus schema roles for the matrix manifest, matrix rows, BPF metadata or BPF SKIP JSON, daemon events, benchmark provenance when applicable, rollback proof, cleanup proof, host refusal proof, privacy scan, static verification logs, audit id, rollback id, VM marker, supported tuple, and attestation status. Manifests must carry an explicit bundle `outcome` derived from matrix rows and protected runner substrate proof. PASS manifests must prove the VM marker was present. `SKIP`, `REFUSE`, and `BLOCKED` manifests may record `benchmark_provenance={"status":"not_applicable",...}` instead of inventing benchmark records when the proof stopped before benchmark artifacts existed. It preserves the same non-release guarantees: `host_mutation=false`, `release_eligible=false`, and `production_capacity_claim=false`.

Evidence-manifest roles are review indexes, not executable instructions: `matrix_manifest` points to the matrix index, `matrix_row` points to each row artifact, `bpf_metadata` or `bpf_skip_json` explains object availability, `daemon_events` and `runtime_samples` point to copied-out VM evidence when present, `rollback_proof`, `cleanup_proof`, `host_refusal_proof`, and `privacy_scan` prove fail-closed invariants for every outcome, and `attestation` records the GitHub artifact attestation result. Missing or unverifiable roles must remain `SKIP`/`REFUSE`/`INCIDENT`; they must not be rewritten into PASS by summary text.

Runner cleanliness remains a companion `runner-cleanliness-proof/v1` artifact validated by `qa/runner_cleanliness_proof_check.py`; it records JIT, ephemeral, or clean-runner proof, no-reuse evidence, runner removal receipt or validator-enforced ephemeral registration receipt, and links to protected-review plus runner-substrate proof instead of adding undocumented fields to `runner-substrate-proof/v1`. Repeated protected bundles are compared with `qa/evidence_bundle_compare_check.py` for expected roles, row set, tuple, and BPF artifact hash only; it does not compare performance metrics or declare regressions.
